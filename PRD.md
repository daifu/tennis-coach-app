## 1. Overview

### 1.1 Product Summary
TennisCoach AI is a cross-platform (web + mobile) application that analyzes a user's tennis video footage, compares their biomechanics against top-10 ATP/WTA professional players, and delivers personalized coaching feedback to help competitive amateur players improve their technique.

### 1.2 Problem Statement
Competitive amateur tennis players lack access to affordable, high-quality biomechanical coaching. Hiring a coach is expensive and time-limited. Existing apps (SwingVision, TennisAI) focus on scoring and ball tracking but do not offer deep technique comparison against specific professional players.

### 1.3 Vision
A personal AI tennis coach that watches your video, compares your swing to your chosen pro, and tells you exactly what to fix — available anytime, on any device.

## 2. Target Users

### 2.1 Primary User
- **Competitive amateur** tennis players (NTRP 3.5–5.5 / ITF 5–8)
- Plays regularly (2–5 times/week)

### 2.2 User Goals

## 4. Features

### 4.1 Core Features (MVP)

#### F1 — Video Upload & Analysis
- User uploads a video of their tennis stroke (serve, forehand, backhand, or volley)
- System detects the player and extracts pose/skeletal keypoints per frame
- System identifies stroke type automatically (or user selects manually as fallback)
- Supported formats: MP4, MOV
- Max video length: 60 seconds (MVP), 5 minutes (post-MVP)
- Processing is asynchronous; user is notified when analysis is complete

#### F2 — Pro Player Comparison
- User selects a reference pro player to compare against
- Initial library: top-10 ATP + top-10 WTA players (20 players total)
- Reference data covers all 4 shot types per player
- System aligns user stroke phases with pro stroke phases using Dynamic Time Warping (DTW)
- Key joint angles and biomechanical metrics are compared side-by-side

#### F3 — Biomechanical Analysis Report
- Per-shot analysis covering:
  - **Serve**: elbow angle at trophy position, shoulder external rotation, knee bend depth, ball toss consistency, hip-shoulder separation at contact
  - **Forehand**: unit turn depth, wrist lag, contact point relative to body, follow-through path
  - **Backhand**: shoulder turn, arm extension, racket drop
  - **Volley**: racket face angle, footwork/split step, punch vs swing ratio
- Similarity score (0–100%) per stroke vs chosen pro
- Visual overlay: user skeleton vs pro skeleton (side-by-side or superimposed)
- Phase-by-phase breakdown (preparation → loading → contact → follow-through)

#### F4 — AI Coaching Feedback
- Natural language feedback generated by LLM (Gemini 2.5 Flash)
- Each piece of feedback follows a strict 3-part format:
  1. **What**: one sentence naming the specific flaw (e.g., "Your elbow drops 15° below the ideal trophy position")
  2. **Why it matters**: one sentence on the performance impact (e.g., "This reduces serve power by limiting shoulder rotation")
  3. **Fix it**: one concrete drill or cue (e.g., "Practice the 'back scratch' drill — hold the trophy position for 2 seconds before each serve rep")
- Maximum 3 flaws highlighted per session — prioritized by impact, not quantity
- No jargon; plain language a club player would hear from a good coach
- Tone: direct, confident, encouraging — no hedging or filler phrases

#### F5 — User Account & History
- Email + password auth (OAuth via Google optional post-MVP)
- Session history: all past uploaded videos and their analysis reports
- Basic progress tracking: similarity score trend over time per shot type

## 5. Technical Architecture

### 5.1 High-Level Stack

```
Frontend (Web):     Next.js 14 (App Router) + Tailwind CSS
Mobile (Post-MVP):  React Native + Expo
Backend API:        FastAPI (Python)
ML Inference:       Modal.com (serverless GPU)
Pose Estimation:    MediaPipe BlazePose (MVP) → YOLOv8-Pose (production)
Temporal Alignment: Dynamic Time Warping (DTW) — C++ core (see §5.5)
LLM Coaching:       Gemini 2.5 Flash API
Database:           PostgreSQL (Supabase)
File Storage:       AWS S3 (video) / Supabase Storage
Auth:               Supabase Auth
Payments:           Stripe
```

### 5.5 Performance-Critical C++ Modules

The following components are implemented in C++ and exposed to Python via `pybind11` bindings, and to the mobile app via native C++ shared libraries:

| Module | Rationale |
|---|---|
| **DTW alignment engine** | Aligns two stroke sequences frame-by-frame; O(n²) time — must run in <50ms on a 120-frame clip |
| **Joint angle calculator** | Per-frame 3D vector math across 33 keypoints × N frames — tight loop, benefits from SIMD |
| **Keypoint normalizer** | Torso-scale normalization applied to every frame of every session |
| **Similarity scorer** | Euclidean distance computation over normalized pose vectors across all phases |

**Toolchain**: C++17, compiled with `-O3 -march=native`; Python bindings via `pybind11`; mobile bindings via JNI (Android) and Objective-C++ (iOS).

**Build strategy**: C++ modules live in `/core/` as a CMake project. Python backend imports them as a compiled `.so`. The frontend never calls C++ directly — only via the API.

### 5.2 Video Processing Pipeline

```
1. User uploads video → S3 → job queued
2. Modal worker picks up job
3. Pose estimation: extract 33 keypoints/frame
4. Stroke phase detection: segment into preparation/loading/contact/follow-through
5. Normalize keypoints by torso scale
6. DTW alignment vs selected pro reference
7. Calculate joint angle deltas per phase
8. Pass structured metrics + key frames → Gemini API
9. Store report in PostgreSQL
10. Notify user (WebSocket / polling)
```

### 5.3 Pro Reference Data

- Source: pose-extracted from publicly available ATP/WTA match broadcast footage
- Stored as: per-player, per-shot-type skeletal keypoint templates (normalized)
- Minimum coverage: 10+ representative stroke samples per player per shot type
- Must be manually reviewed and labeled before inclusion

### 5.4 Performance Requirements

| Metric | Target |
|---|---|
| Video upload (50MB) | < 5s |
| Analysis turnaround (Free tier) | < 90 seconds |
| Analysis turnaround (Pro tier) | < 30 seconds |
| DTW alignment (120-frame clip) | < 50ms (C++ module) |
| Joint angle calculation (full session) | < 20ms (C++ module) |
| Web app initial load | < 1.5s |
| Report page load (cached) | < 300ms |
| API uptime | 99.5% |

## 6. UX Requirements

### 6.1 Key User Flows

**Flow 1 — First-Time Analysis**
1. Sign up → land on dashboard
2. Tap "Analyze My Stroke"
3. Select shot type
4. Select pro player to compare against
5. Upload video (drag-and-drop or file picker)
6. See processing screen with progress indicator
7. Receive report: score, visual overlay, written feedback, drills

**Flow 2 — Returning User**
1. Log in → dashboard shows history + progress charts
2. Upload new video or review past reports
3. Compare current vs past similarity scores

### 6.2 Camera Guidance
- In-app guide shows optimal camera placement before recording
  - Side-on view, camera at hip height, 10–15 feet from player
- Warning shown if uploaded video appears to be filmed from a poor angle

### 6.3 Accessibility
- WCAG 2.1 AA compliance
- Video player with playback controls accessible via keyboard

---

## 7. Non-Functional Requirements

- **Privacy**: User videos are stored encrypted at rest; never used for training without explicit consent
- **Data retention**: Videos deleted after 90 days (Free) / 1 year (Pro) unless user deletes sooner
- **GDPR / CCPA**: Compliant data deletion on request
- **Security**: Auth via Supabase JWT; no raw passwords stored; HTTPS only
- **Scalability**: Stateless API; GPU workers scale horizontally on Modal

## 10. Decisions & Open Questions

### 10.1 Resolved Decisions

| # | Decision | Rationale |
|---|---|---|
| D1 | **Stroke type auto-detection is post-MVP; manual selection in MVP** | Reduces ML scope significantly; auto-detection requires a separate classification model. Manual selection is fast for the user and removes a potential source of errors in the MVP. |
| D2 | **Monthly-only Stripe billing at launch** | Simpler to implement and test; annual plan adds proration/refund complexity. Add annual option in Phase 2 once retention data justifies it. |
| D3 | **No public launch waitlist — open beta immediately** | Solo project; waitlist management adds overhead with minimal benefit at early stage. Launch openly and iterate on real user feedback. |

### 10.2 Open Questions

- [ ] How to handle copyright on pro player reference footage used for pose extraction? (Consider: extract pose only, never redistribute raw video; consult a lawyer before scaling)
- [ ] Is DTW sufficient for temporal alignment, or do we need a learned stroke segmentation model? (Start with DTW; evaluate accuracy after 100+ real user sessions)
